# Patch Notes — CAF Versioning Foundations (v1)

Date: 2026-03-04

## Goals

- Introduce a single source-of-truth CAF version.
- Stamp CAF-produced artifacts (starting with feedback packets + key CAF meta reports) with the producing CAF version.
- Keep changes deterministic and drift-resistant; no semantic heuristics.

## Added

- `CAF_VERSION` (repo root): single source of truth (initial value `0.1.0`).
- `VERSIONING.md` (repo root): normative SemVer + stamping policy.
- `tools/caf/lib_caf_version_v1.mjs`: best-effort reader + stamp helpers:
  - `readCafVersionSync()`
  - `cafMarkdownStampLine(toolLabel)`
  - `cafBulletStampLine()`

## Updated (stamping)

### Feedback packets
All CAF scripts that write `reference_architectures/<instance>/feedback_packets/BP-*.md` now include:

- `- Generated By: CAF vX.Y.Z`

Inserted immediately after `- Date: ...`.

### CAF meta reports (markdown)
These scripts now prefix their generated markdown outputs with:

- `<!-- Generated by CAF vX.Y.Z -->`

(Only if the stamp is not already present.)

- `tools/caf/worker_traceability_mindmap_v3.mjs`
- `tools/caf/build_retrieval_debug_v1.mjs`
- `tools/caf/build_candidate_selection_report_v1.mjs`
- `tools/caf/build_retrieval_context_blob_v1.mjs`

## Notes / non-goals (this patch)

- This does **not** yet stamp all machine-readable artifacts (e.g., `.jsonl`) because in-band stamping could break parsers.
  - Normative direction (future): emit `<artifact>.cafmeta.json` sidecars when in-band stamping is unsafe.
- No `/caf-version` skill was added in this patch.

## Suggested next steps

1) Decide the minimal "machine-readable provenance" approach:
   - Sidecar metadata (`.cafmeta.json`) for `.jsonl` and strict JSON surfaces.

2) Add a maintainer command (likely under `/caf-meta`) to propose the next SemVer bump
   without running git inside CAF workflows. Recommended shape:
   - deterministic change report generator (maintainer-run, can use git)
   - a pure evaluator that maps the report to a bump + rationale

3) Consider adding a small compatibility test:
   - verify stamped markdown still parses where expected (mermaid fences, etc.)
   - verify feedback packet parser remains unaffected (it reads only Severity)
