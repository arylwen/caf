# Contura CAF — Official Derivation Process (Layers 6–8) (v1)

Layers **1–5 are treated as locked inputs**. If a missing requirement→choice or choice→validation/evidence mapping is discovered, the derivation must fail closed and emit a feedback packet.

---

## 0. Roles and core rule

### Architect (chooser)

- Makes **only** choices that are explicitly offered by the architecture library (templates + parameters + allowed values).
- Does **not** invent new degrees of freedom during derivation.
- If a choice is required but not offered or not mappable, the architect stops and escalates via a feedback packet.

### Smart Assistant (transformer)

- Generates derived documents **only** when additional architect decisions are not required.
- Must refuse to guess. If derivation requires missing “glue,” it must emit a feedback packet.

### Validator (gatekeeper)

- Produces PASS/FAIL outcomes for each validation gate.
- On FAIL, emits a feedback packet and stops the derivation (fail-closed).

## 3. Core derivation steps (Layers 6–8)

This section defines the **core** derivation loop that produces:
- pinned architecture shape parameters (Layer 6)
- pinned profile parameters + derived rails (Layer 8)
- merge-safe system/application specifications (Layer 6)

It does **not** produce Layer 7 documentation bundles.

### Step A1 — Create instance + pin architecture shape parameters (Architect chooses)

**Inputs**
- `07_contura_parameterized_architecture_templates_v1.md`

**Architect outputs**
- `reference_architectures/<name>/layer_6/architecture_shape_parameters.yaml`

**Validation Gate V1 — Pin Closure (fail-closed)**
Validator checks:
- all instantiated templates are declared (with version)
- every exposed parameter is pinned
- pinned values are members of the allowed value sets
- no placeholder values (TBD/TODO/UNKNOWN/empty)

On FAIL:
- Emit feedback packet: `reference_architectures/<name>/feedback_packets/BP-YYYYMMDD-pin-closure.md`
- Stop.

---

### Step A2 — Pin Phase 8 profile parameters (Architect chooses)

**Architect outputs**
- `reference_architectures/<name>/layer_8/profile_parameters.yaml`

The architect pins **exactly three knobs**:
- `lifecycle.evolution_stage`
- `lifecycle.generation_phase`
- `platform pins (infra_target/packaging/runtime_language/database_engine)`

No other decision knobs are allowed in the architect-authored UX file.

**Validation Gate V2 — Profile Parameters Schema (fail-closed)**
Validator checks:
- profile parameters conform to `architecture_library/phase_8/84_phase_8_profile_parameters_schema_v1.yaml`
- exactly the allowed knobs are present (plus permitted metadata)

On FAIL:
- Emit feedback packet: `reference_architectures/<name>/feedback_packets/BP-YYYYMMDD-profile-parameters-invalid.md`
- Stop.

---

### Step A3 — Derive Layer 8 rails + generate merge-safe specs (CAF + validator)

This step is executed by `/caf-arch` when `lifecycle.generation_phase == architecture_scaffolding`.

**Inputs**
- `reference_architectures/<name>/layer_6/architecture_shape_parameters.yaml`
- `reference_architectures/<name>/layer_8/profile_parameters.yaml`
- `architecture_library/phase_8/90_phase_8_profile_derivation_policy_matrix_v1.yaml` (rails derivation)
- `skills/caf-system-architect/SKILL.md` (spec generation)

**CAF generates (no new choices)**
- `reference_architectures/<name>/layer_8/profile_parameters_resolved.yaml`
- `reference_architectures/<name>/spec/guardrails/syntactic_checks.md` (optional)
- `reference_architectures/<name>/layer_6/system_spec_v1.md`
- `reference_architectures/<name>/layer_6/application_spec_v1.md`

Notes:
- Specs are **merge-safe**: CAF updates CAF-managed blocks only; architect-edit blocks preserve human content.
- CAF may seed a removable sample domain (Widget) on first creation of `application_spec_v1.md`.
- CAF MUST record missing semantics as open questions; it MUST NOT guess.

**Validation Gate V3 — Architecture Scaffolding Readiness (fail-closed)**
Validator checks (structural):
- Layer 8 resolved view exists
- system and application specs exist and contain required managed/edit blocks
- no placeholder values in CAF-managed blocks

On FAIL:
- Emit feedback packet: `reference_architectures/<name>/feedback_packets/BP-YYYYMMDD-arch-scaffolding-fail.md`
- Stop.

## 4. Feedback packets (fail-closed escalation)

### When to emit

Emit a feedback packet immediately when:
- a required choice is not offered by CAF (Missing option)
- a required mapping is not present (Missing mapping / missing glue)
- library standards/specs/examples conflict (Spec inconsistency)
- generation would require guessing

### Required fields (minimum)

Store at: `reference_architectures/<name>/feedback_packets/BP-YYYYMMDD-<slug>.md`

Include:
- **Stuck At:** step + file path + template/parameter involved
- **Required Capability:** one sentence
- **Observed Constraint:** why offered choices/mappings are insufficient
- **Gap Type:** Missing option | Missing mapping | Spec inconsistency
- **Minimal Fix Proposal:** smallest library change that unblocks derivation
- **Evidence:** which CAF docs/sections were consulted

## 5. Glue maintenance loop (improve grounding and traceability)

This is an ongoing maintenance loop that improves CAF so:
- advisory decision candidates can be grounded (pins/prose → patterns), and
- validation/evidence expectations can be traced (patterns/rails → checks/evidence).

### Step B1 — Pins/prose → pattern candidates (semantic retrieval surface)

Inputs:
- pinned pins + prose descriptions
- CAF pattern definitions + index

Output:
- a stable retrieval surface and schemas (index/playbooks) that allow grounded advisory candidate generation without deterministic string matching.

### Step B2 — Pattern candidates → validation/evidence anchors

Inputs:
- CAF patterns and decision checklist anchors
- validation standards/checklists
- evidence expectations standards

Output:
- mappings that allow validators/evaluators to check “this candidate was grounded” and “this requirement has an anchor.”

## 6. Documentation / audit loop (Layer 7 + architecture document)

The following steps are **not part of the core `/caf-arch` derivation loop**. They exist to produce documentation and audit/traceability bundles without introducing new choices.

### Step D1 — Generate Layer 7 derived bundle (no new choices)

- Tooling: `caf-layer7` (and `caf-arch-doc` may call it cache-aware).
- Outputs (if applicable to pinned templates):
  - `reference_architectures/<name>/layer_7/README.md`
  - `reference_architectures/<name>/layer_7/adrs/*`
  - `reference_architectures/<name>/layer_7/validation_mapping/validation_mapping.md`

On any missing glue mappings or ambiguous requirements, this step MUST fail closed and emit a feedback packet.

### Step D2 — Generate architecture documentation (merge-safe)

- Tooling: `caf-arch-doc`
- Inputs: Layer 6 specs/designs + Layer 7 bundle (generated/ensured by the doc loop) + Layer 8 rails
- Output: companion repo architecture documentation artifacts
