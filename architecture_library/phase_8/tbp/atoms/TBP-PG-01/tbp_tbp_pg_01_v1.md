TBP_ID: TBP-PG-01
NAME: PostgreSQL persistence binding
INTENT: Bind PST-01 roles to PostgreSQL-specific service/env/runtime wiring while complementary ORM TBPs own ORM realization details when selected.
SCOPE: Application
APPLIES_WHEN: database.engine == postgresql
EXTENDS_CORE_PATTERNS: PST-01; CFG-01
BINDS_MODULE_ROLES: repository_ports; persistence_adapters; persistence_models; config_model; config_provider

ROLE_BINDINGS:

NOTE (scaling): TBP-PG-01 should NOT hardcode a single resource (e.g., widget). It provides global Postgres wiring + a reusable adapter surface; per-resource repositories are generated by persistence_implementation tasks using the domain model.
NOTE (ownership): When `persistence.orm == sqlalchemy_orm`, TBP-SQLALCHEMY-01 owns SQLAlchemy runtime/metadata/bootstrap realization and the accepted `DATABASE_URL` contract shape; TBP-PG-01 remains the database engine wiring/config contract and any engine-specific adapter hook exposed to the application boundary.
- repository_ports: Defined as Python interfaces/protocols used by domain/application layers to access persistence (e.g., `{Resource}Repository`).
- persistence_adapters: Concrete adapters using a PostgreSQL driver/ORM in the infrastructure layer (e.g., `Postgres{Resource}Repository`).
- persistence_models: Persistence-level models/schema representations (ORM models, table metadata, or explicit row mappers), distinct from domain models when applicable.
- config_model: Includes DB connection settings (e.g., `DATABASE_URL`, `DB_HOST`, `DB_USER`, etc.).
- config_provider: Loads DB settings from env and provides them to the persistence_adapters.

ADDS_EVIDENCE_HOOKS:
- E-TBP-PG-01-01: Dependency spec includes a PostgreSQL client/driver (e.g., `psycopg`/`psycopg2` OR `asyncpg`).
- E-TBP-PG-01-02: Configuration references PostgreSQL through engine-owned env/service wiring (for example `DATABASE_URL` and/or `DB_*` variables) without taking ownership of ORM-specific URL syntax validation.
- E-TBP-PG-01-03: Presence of migration artifacts (e.g., `alembic/` with `versions/`, or Django `migrations/`).
- E-TBP-PG-01-04: If using Compose, compose declares a `postgres` image service.
- E-TBP-PG-01-05: When AP/CP startup depends on compose-materialized `postgres`, compose defines a postgres `healthcheck:` and dependent services use `depends_on.postgres.condition: service_healthy`.

ADDS_STRUCTURAL_VALIDATIONS:
- V-TBP-PG-01-01: DB credentials/host/URL are not hard-coded in source; provided via config_provider/env.
- V-TBP-PG-01-02: A migrations mechanism exists in-repo (at least one migration artifact is committed).
- V-TBP-PG-01-03: persistence_adapters encapsulate SQL execution; callers outside infrastructure_adapters do not issue SQL directly.
- V-TBP-PG-01-04: Compose-backed local startup does not rely on postgres container creation order alone when AP/CP bootstrap touches the database during startup.
