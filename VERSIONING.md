# CAF Versioning (Normative)

## Source of truth

- Repo root `VERSION`

CAF uses **Semantic Versioning**: `MAJOR.MINOR.PATCH`.

## What is versioned

- **CAF runtime version**: the behavior of `/caf` and `/caf-meta` workflows, deterministic helpers under `tools/caf/**`, and the normative pattern library under `architecture_library/**`.
- **Artifacts produced by CAF**: each CAF-produced artifact should include a **CAF version stamp** indicating the CAF version that produced it.

## SemVer policy

Increment:

### MAJOR

Use when a change is *breaking* for users or CAF tooling, including (non-exhaustive):

- Breaking changes to **contracts** under `tools/caf/contracts/**`
- Breaking changes to the **instance layout** or required files
- Breaking changes to **pattern schema** (e.g., `caf.kind`, option-set structure) that invalidate existing patterns or instances
- Breaking changes to artifact formats that downstream scripts/agents rely on

### MINOR

Use for backward-compatible capability expansion, including:

- New workflow steps, new gates, new deterministic helpers
- New pattern families/types that do not invalidate existing instances
- Additive changes to artifact formats (new sections/fields) that remain compatible

### PATCH

Use for backward-compatible fixes, including:

- Bug fixes in deterministic helpers
- Tightening validations that correct obvious invalid cases without invalidating valid instances
- Documentation fixes

## Version stamp policy (artifacts)

- Human-readable artifacts (`.md`, `.txt`) include a top-of-file stamp:
  - `<!-- Generated by CAF vX.Y.Z -->`
- Feedback packets include a stamp line in the packet header:
  - `- Generated By: CAF vX.Y.Z`
- Machine-readable artifacts should not be made invalid by stamping.
  - If an artifact format cannot safely embed a stamp (e.g., `.jsonl`), prefer a **sidecar metadata file** in the future (e.g., `<artifact>.cafmeta.json`).
