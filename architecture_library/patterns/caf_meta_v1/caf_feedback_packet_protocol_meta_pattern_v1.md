# CAF Feedback Packet Protocol Meta-Pattern (v1)

This meta-pattern defines the **normative** protocol for CAF feedback packets.
It is intended to be:

- **authoritative** (single source of truth for packet structure + lifecycle),
- **drift-resistant** (tools/skills can reference it, audits can enforce it),
- **deterministic** (Node-script-friendly parsing and upgrade behavior).

Feedback packets are the canonical fail-closed escalation artifact.
They must be written to disk (never printed inline in chat output).

---

## 1) Scope

This protocol applies to **all CAF-emitted feedback packets**, regardless of producer:

- deterministic tools (gates, linters, mergers, indexers)
- semantic steps (assistants / agent workflows)
- maintainers resolving drift manually

Typical storage location:

- `reference_architectures/<name>/feedback_packets/*.md`

---

## 2) Status lifecycle (normative)

Every feedback packet MUST be in exactly one status:

1. `pending` — when the packet is created.
2. `stale` — when a `/caf` command starts, all *existing* `pending` packets MUST be marked `stale`.
3. `resolved` — when the issue is fixed. A resolved packet MUST NOT be deleted.

### Resolution rule (normative)

When a deterministic gate or scripted helper can prove that the blocking condition no longer exists, it SHOULD mark its own matching packet(s) `Status: resolved` automatically.
Manual packet-status edits by agents or human operators are allowed only as an exceptional fallback and MUST NOT be the primary recovery path.

### Command start rule (normative)

At the beginning of any `/caf <command>` run:

1. Enumerate all feedback packets under the target instance’s `feedback_packets/` directory.
2. For each packet with `Status: pending`, update it in-place to `Status: stale`.

**Purpose:** any `pending` packet present after command start was created during this command run.

---

## 3) Canonical header schema (normative)

Each feedback packet MUST start with a title line and a required header block.

### Title

The first line MUST be a Markdown H1:

```md
# Feedback Packet: <PACKET_ID>
```

`<PACKET_ID>` is producer-defined (commonly `BP-YYYYMMDD-<slug>`), but MUST be stable for the packet.

### Required header lines

Immediately after the title, include these lines **in this exact order**:

```md
- Date: YYYY-MM-DD
- Generated By: CAF vX.Y.Z
- Status: pending|stale|resolved
- Severity: blocker|advisory
```

#### Parsing rules (deterministic)

- Each required line MUST be present exactly once.
- Keys are case-sensitive and MUST match exactly: `Date`, `Generated By`, `Status`, `Severity`.
- The separator MUST be `: ` (colon + single space).
- The value MUST be on the same line.

#### Version stamping (normative)

- `Generated By` MUST contain **only the CAF version**, in the form `CAF vX.Y.Z`.
- It MUST NOT include tool names, file paths, or command names.
- Producers SHOULD read the version from the repo’s `VERSION` file.
- If the version cannot be determined deterministically, producers MUST emit: `CAF v0.0.0+unknown`.

#### Additional header fields (optional)

Additional `- Key: Value` lines MAY appear, but MUST come **after** the required four lines.

Preferred optional fields when applicable:

- `- Instance: <name>`
- `- Stuck At: <tool_or_skill_path>`
- `- Gap Type: <short classification>`
- `- Issue Tracker: <url>` — use when the packet explicitly recommends strengthening a reusable framework/provider/gate seam so operators know where to file or update the follow-up.

---

## 4) Canonical body sections (normative)

Each packet MUST include these sections (headings must match exactly):

```md
## Minimal Fix Proposal

## Evidence

## Autonomous agent guidance

## Human operator guidance
```

### Section semantics (normative)

- **Minimal Fix Proposal** MUST state the smallest change that unblocks the workflow.
- **Evidence** MUST list concrete file paths and locators (headings, YAML paths, line ranges, block ids).
- **Autonomous agent guidance** MUST specify deterministic remediation steps:
  - concrete file edits (or deletions),
  - and whether the agent should continue in-place or rerun an exact `/caf` command.
- **Human operator guidance** MUST specify the recovery path for a human operator:
  - if reset + rerun is the correct recovery, name the exact reset helper and the exact `/caf` command,
  - if a packet is advisory-only, name the exact diagnostic script commands and state that they are advisory.

---

## 5) Gate rules (normative)

Gates MUST interpret feedback packets using the following rules:

1. `Status: resolved` packets **never block** any gate.
2. `Status: pending|stale` packets block **only** if `Severity: blocker`.
3. `Severity: advisory` packets never block, but MAY be surfaced in reports.

---

## 6) Legacy packet handling (normative)

Packets missing required fields MUST be treated as **legacy** and handled deterministically.

### Defaults

On the next command start:

- Default `Status` to `stale`.
- Default `Severity` to `blocker` **unless** the packet explicitly declares `- Severity: advisory`.

### Upgrade behavior

At command start, the command runner MUST upgrade legacy packets by inserting the required header lines
immediately after the title, using the defaults above.

If the legacy packet already has some required lines:

- preserve existing values when valid,
- insert only the missing required lines,
- ensure final required line order matches the canonical schema.

---

## 7) Audit expectations (recommended, drift-resistant)

Maintainers and audit tools SHOULD validate:

- required header lines are present and ordered
- values are from the allowed enums
- required body headings exist
- `Generated By` contains only a CAF version token
- no packet is deleted when resolved (status flip only)

On audit failure, the auditor MUST fail closed and emit a feedback packet describing the non-conformance.
